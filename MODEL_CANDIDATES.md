# Маленькие мультиязычные модели для lay

Дата: 2026-05-01.

Цель: держать короткий список моделей, которые по очереди тестируются как
локальный LLM-арбитр для `lay`. Для боевого режима важнее всего задержка на
коротком A/B-ответе, стабильность на русском/английском и отсутствие лишней
генерации.

## Русские модели для двух слов

Добавлен отдельный стенд:

```text
eval/two_word_cases.tsv
scripts/eval_two_word_models.py
```

Вывод по сгенерированному стресс-набору на 3118 двухсловных кейсов:

- генератор кандидатов покрывает 3118/3118 ожидаемых ответов;
- простой словарно-раскладочный scorer без моделей даёт 3118/3118;
- `cointegrated/rubert-tiny2` как raw masked-LM scorer провалил первые 20
  смешанных кейсов: 0/20;
- `ai-forever/sage-fredt5-distilled-95m` как raw generator тоже провалил первые
  20 смешанных кейсов: 0/20;
- guarded `rubert-tiny2` arbiter дал 3108/3118;
- guarded `sage-fredt5` arbiter дал 3115/3118;
- guarded consensus arbiter дал 3115/3118.

Вывод: эти модели нельзя пускать напрямую вместо `lay`. Их можно продолжать
тестировать только как осторожные слои вокруг уже построенных кандидатов:
`lay` строит варианты, модель даёт голос, решение остаётся за deterministic
контуром. На текущем большом стенде даже guarded-арбитры ухудшают результат
относительно deterministic scorer.

## Сильный тест LFM2.5

Набор: 40 A/B-кейсов для `lay`: неправильная русская раскладка, обратная
перекладка в английский, корректный русский/английский текст, домены, email,
URL, смешанные токены, короткие одиночные буквы.

| Режим         | Точность | Ошибки | Средн. мс | Медиана мс | Итог                      |
|---------------|---------:|-------:|----------:|-----------:|---------------------------|
| current_raw   |    42.5% |  23/40 |      1040 |       1043 | не подходит               |
| strict_en_raw |    57.5% |  17/40 |      1635 |       1633 | лучше, но всё равно плохо |

Вывод: LFM2.5 не годится как безопасный быстрый арбитр для `lay`. В текущем
промпте она почти всегда оставляет оригинал, а в более строгом промпте почти
всегда выбирает перекладку. Оба перекоса опасны для боевого режима.

## BitNet

BitNet интересна отдельно от Ollama-кандидатов: это не обычный GGUF-путь через
llama.cpp, а CPU-native runtime `bitnet.cpp` с 1.58-bit весами и специальными
ядрами. Для `lay` это потенциально полезно именно как быстрый короткий
классификатор, но русский и устойчивость A/B-выбора надо проверять отдельно.

| Пункт     | Значение                                     |
|-----------|----------------------------------------------|
| Runtime   | bitnet.cpp, не Ollama                        |
| CPU-смысл | 1.58-bit веса, быстрые ternary kernels       |
| Модель    | microsoft/BitNet-b1.58-2B-4T-gguf            |
| Параметры | 2.4B                                         |
| Квант     | i2_s для x86                                 |
| Риск      | русский и A/B-стабильность надо проверять    |
| Статус    | установлена в ~/projects/BitNet              |
| Вердикт   | чистый prompt слабый, гибрид перспективный   |

Черновые команды установки:

```bash
cd ~/projects
git clone --recursive https://github.com/microsoft/BitNet.git
cd BitNet

python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
pip install huggingface_hub

huggingface-cli download microsoft/BitNet-b1.58-2B-4T-gguf \
  --local-dir models/BitNet-b1.58-2B-4T

python setup_env.py -md models/BitNet-b1.58-2B-4T -q i2_s
```

Локальная сборка на ThinkPad T480 потребовала один const-correctness патч в
`src/ggml-bitnet-mad.cpp`: `int8_t * y_col` -> `const int8_t * y_col`.

### Сильный тест BitNet

Запуск: `llama-server`, порт `18081`, `i2_s`, 4 CPU-потока, `ctx=512`,
grammar `A|B`, `cache_prompt=true`. Набор тот же: 40 A/B-кейсов для `lay`.

| Режим                  | Точность | Ошибки | Средн. мс | Медиана мс | p95 мс | Итог                        |
|------------------------|---------:|-------:|----------:|-----------:|-------:|-----------------------------|
| llama-server + grammar |    65.0% |  14/40 |       176 |        169 |    245 | быстро, но качество среднее |

Вывод: BitNet уже интересна по скорости, если держать `llama-server` горячим и
использовать `cache_prompt=true`. Как чистый A/B-арбитр она пока небезопасна:
часто выбирает перекладку даже для нормального текста. Её можно продолжать
тестировать только вместе с детерминированными защитными правилами.

### Расширенный тест BitNet

Проверены разные prompt-стратегии на том же наборе из 40 кейсов. Все варианты
использовали горячий `llama-server`, grammar `A|B`, `cache_prompt=true`.

| Стратегия             | Точность | Ошибки | Средн. мс | Медиана мс | Итог                    |
|-----------------------|---------:|-------:|----------:|-----------:|-------------------------|
| balanced              |    65.0% |  14/40 |       198 |        179 | лучший чистый prompt    |
| safe_keep             |    52.5% |  19/40 |       227 |        184 | слишком часто держит A  |
| ru_safe               |    55.0% |  18/40 |       219 |        182 | тоже перекос в A        |
| compact_safe          |    60.0% |  16/40 |       225 |        195 | портит нормальный текст |
| more_keep_examples    |    55.0% |  18/40 |       250 |        193 | качество хуже           |
| rules + BitNet        |    85.0% |   6/40 |       181 |        179 | без словарей            |
| dict + rules + BitNet |    95.0% |   2/40 |       194 |        193 | лучший вариант          |

Лучший результат получился не от prompt-а, а от архитектуры:

1. Сначала структурные правила `lay`: URL/domain/email/acronym, смешанные
   токены, одиночные буквы, латинские островки.
2. Затем системные словари `/usr/share/hunspell/ru_RU.dic` и
   `/usr/share/hunspell/en_US.dic`.
3. Только оставшиеся спорные случаи уходят в BitNet.

| Метрика             | Значение |
|---------------------|---------:|
| Всего кейсов        |       40 |
| Верно               |       38 |
| Ошибок              |        2 |
| Точность            |    95.0% |
| Вызовов BitNet      |        8 |
| Решено правилами    |       32 |
| Средн. LLM latency  |   194 мс |
| Медиана LLM latency |   193 мс |

Вывод: BitNet нельзя использовать как единственный судья. Но как последний
быстрый CPU-backend после правил и словарей она уже выглядит рабочим кандидатом.
